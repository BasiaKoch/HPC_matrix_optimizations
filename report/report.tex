\begin{abstract}
An in-place Cholesky factorization routine $C = LL^T$ was implemented in C and
optimised in several stages for a single 76-core CSD3 icelake node.

Correctness was verified using exact small examples, reconstruction residuals,
output-layout checks, and consistency across thread counts.
Compiler optimisations and simple loop transformations improved single-thread
performance by around 13--16$\times$ compared to an \texttt{-O0} baseline.

A straightforward OpenMP parallelisation achieves good performance at moderate
thread counts, but performance decreases at higher thread counts, likely due to
synchronisation overhead.

To address this, the algorithm was restructured into a panel-blocked
(right-looking) form with panel width $\mathrm{NB}=96$. This reduces the number
of global synchronisation steps and improves scalability. The blocked version
achieves a speedup of 59.5$\times$ and a parallel efficiency of 78.2\% at
76 threads for $n=8000$.

Further optimisations aimed at improving data locality and instruction-level
parallelism (including column packing, per-thread panel storage, and loop
unrolling) improved single-thread performance by a factor of 2.52 compared to
the initial blocked version. A round-robin scheduling strategy was also used
to improve load balance in the triangular update phase.

The final implementation reaches 182.7~GFLOP/s at 76 threads for $n=8000$,
representing a 24\% improvement over the previous version.
\end{abstract}

\section{Introduction}

Cholesky factorization $C = LL^T$ is widely used in applications such as
Gaussian process regression, Bayesian inference, and numerical optimisation.
In these settings, it is often required to compute both the inverse and the
determinant of a symmetric positive-definite covariance matrix.

Given a covariance matrix $C \in \mathbb{R}^{n \times n}$ with entries
\[
C_{ij} = \langle (x_i - \langle x_i \rangle)(x_j - \langle x_j \rangle) \rangle,
\]
the log-determinant can be computed from the Cholesky factor as
\begin{equation}
  \log |C| = 2 \sum_{p=0}^{n-1} \log L_{pp}.
\end{equation}

The computational cost of the factorisation is $O(n^3)$, so performance is
critical for large matrices. At the same time, numerical correctness must be
maintained.

This report describes the development and optimisation of an OpenMP-based
Cholesky factorisation routine on a single CSD3 icelake node.
The final implementation achieves 182.7~GFLOP/s at 76 threads for $n=8000$.

For smaller matrices (e.g.\ $n=2000$), higher throughput is observed
(238.4~GFLOP/s at 76 threads), which is consistent with better cache utilisation.
For larger matrices, performance decreases, which is likely due to increased
pressure on the memory system.

\section{Algorithm and Interface}

\subsection{Cholesky recurrence}

The factorisation proceeds column by column for $p = 0, \ldots, n-1$.
At each step, the diagonal element is computed as
\begin{equation}
  L_{pp} = \sqrt{C_{pp} - \sum_{s=0}^{p-1} L_{ps}^2},
\end{equation}
and the entries below the diagonal are given by
\begin{equation}
  L_{ip} = \frac{C_{ip} - \sum_{s=0}^{p-1} L_{is} L_{ps}}{L_{pp}}, \quad i > p.
\end{equation}

These steps must be performed sequentially, as each column depends on the
previous ones. However, the update of the remaining submatrix,
\[
C_{ij} \leftarrow C_{ij} - L_{ip} L_{jp}, \quad i,j > p,
\]
can be performed independently across rows and is therefore suitable for
parallelisation.

\subsection{Function interface}

The function
\begin{verbatim}
double mphil_dis_cholesky(double *c, int n);
\end{verbatim}
takes a pointer to a row-major $n \times n$ matrix and performs the
factorisation in place.

On return:
\begin{itemize}
  \item the lower triangle contains $L$,
  \item the upper triangle contains $L^T$.
\end{itemize}

The function returns the wall-clock time in seconds. Inputs with
$n < 1$ or $n > 100000$ are rejected.

\subsection{Correctness verification}

Correctness was assessed using several complementary checks.
The main criterion is the reconstruction error of the factorisation,
rather than the log-determinant alone.

The test suite was run on a CSD3 icelake node for both:
\begin{itemize}
  \item a \textbf{strict build} (without \texttt{-ffast-math}),
  \item a \textbf{performance build} (with \texttt{-ffast-math}),
\end{itemize}
and across multiple thread counts ($T = 1, 2, 4, 8, 76$).
All tests passed in all configurations.

\begin{enumerate}
  \item \textbf{Known-$L$ tests.}
  Matrices of the form $C = L_{\mathrm{ref}} L_{\mathrm{ref}}^T$ were generated,
  and the computed factor was compared directly to $L_{\mathrm{ref}}$.
  Test sizes included values around block boundaries
  ($n = 95, 96, 97$, etc.), where indexing errors are more likely.

  The maximum observed errors were small:
  \[
  \max |L - L_{\mathrm{ref}}| \approx 10^{-14}, \quad
  \|A - LL^T\|_F / \|A\|_F \approx 10^{-16},
  \]
  which is consistent with double precision accuracy.

  \item \textbf{Output layout.}
  The symmetry between the lower and upper triangles was checked and found
  to hold exactly in all cases.

  \item \textbf{Boundary cases.}
  The implementation correctly handles $n=1$ and rejects invalid sizes.

  \item \textbf{Thread consistency.}
  Results obtained with multiple threads were compared to the single-thread
  result. No differences were observed, indicating correct synchronisation.
\end{enumerate}

The log-determinant was used as a secondary check only. While it provides a
useful scalar summary, it does not detect all possible errors (for example,
incorrect off-diagonal entries).

\section{Optimisation Strategy}

\subsection{v1 --- Baseline (tag \texttt{v0.1-baseline})}

The baseline implementation follows the provided algorithm and is compiled
with \texttt{-O0}. The trailing update uses a loop ordering where
\texttt{c[i*n+j]} is accessed with a large stride in the inner loop.
For large $n$, this leads to poor cache utilisation and limits performance.

\subsection{v2 --- Serial optimisation (tag \texttt{v0.2-serial-opt})}

Two simple changes are made to improve single-thread performance:

\begin{itemize}
  \item \textbf{Loop interchange.}
  The inner loop is changed to iterate over $j$, making
  \texttt{c[i*n+j]} contiguous in memory. This improves cache locality and
  allows the compiler to vectorise the loop more effectively.

  \item \textbf{Loop-invariant hoisting.}
  The value \texttt{c[i*n+p]} is reused inside the inner loop, so it is
  loaded once per row instead of once per iteration.
\end{itemize}

With compiler optimisations enabled (\texttt{-O3 -march=native -ffast-math}),
this version is significantly faster than the baseline.

\subsection{v3\_openmp --- Flat OpenMP parallelism (tag \texttt{v0.3-openmp-v1})}

The trailing update is parallelised over rows using
\texttt{\#pragma omp for}. Since each row is independent, this is a natural
choice for parallelisation.

A single \texttt{omp parallel} region is used around the outer loop to avoid
repeated thread creation. Within each iteration, \texttt{omp single} is used
for the diagonal and normalisation steps, followed by a barrier before the
parallel update.

This approach improves performance at moderate thread counts. However, for
$n=2000$, performance peaks at around 48 threads and then decreases at higher
thread counts. This behaviour is consistent with the cost of frequent
synchronisation becoming significant when the amount of work per thread is
small.

\subsection{v4\_openmp\_blocked --- Panel-blocked algorithm}

To reduce synchronisation overhead, the algorithm is restructured to process
blocks of columns instead of a single column at a time.

The matrix is processed in panels of width $\mathrm{NB}$. For each panel,
three steps are performed:

\begin{itemize}
  \item \textbf{Phase 1 (serial):} factorise the diagonal block.
  \item \textbf{Phase 2 (parallel):} solve for the block below the panel.
  \item \textbf{Phase 3 (parallel):} update the trailing submatrix.
\end{itemize}

This reduces the number of global synchronisation points from $O(n)$ to
$O(n/\mathrm{NB})$, which improves scalability.

The panel width was chosen empirically. Values in the range
$\mathrm{NB}=64$--$256$ give similar performance, with $\mathrm{NB}=96$
providing a good balance across thread counts. Larger panels reduce the number
of synchronisation steps, but may reduce cache efficiency. The optimal value
is therefore not unique, and $\mathrm{NB}=96$ is used as a consistent choice
for the main experiments.

\subsection{v5\_openmp\_blocked --- Further optimisation}

Several additional optimisations are introduced on top of v4. These aim to
improve memory access patterns and increase instruction-level parallelism.

\begin{itemize}
  \item \textbf{Column packing (Phase 1).}
  The column used in the panel update is copied into a contiguous buffer.
  This avoids repeated long-stride accesses and improves locality.

  \item \textbf{Per-thread packed panel (Phase 2).}
  Each thread copies the current panel block into a private buffer.
  This reduces repeated access to widely spaced memory locations during the
  triangular solve.

  \item \textbf{Loop unrolling (Phase 3).}
  The inner update loop is unrolled to process multiple columns at once.
  This increases the amount of independent work available to the processor.

  \item \textbf{Scheduling strategy.}
  \texttt{schedule(static,1)} is used in the triangular update to distribute
  work more evenly across threads, as the amount of work per row increases
  with the row index.
\end{itemize}

These changes improve performance, although their individual contributions
are not isolated in this report. The final version shows a clear improvement
over v4 in both single-thread and multi-thread performance.

\section{Experimental Setup}

\paragraph{Platform.}
All experiments were run on a single node of the CSD3 icelake partition
(Intel Xeon Platinum 8368Q, 2 sockets × 38 cores, 76 cores total).
The node consists of two NUMA domains, with private L1 and L2 caches per core
and a shared L3 cache per socket.

Jobs were submitted using SLURM, typically requesting one full node.

\paragraph{Compilation.}
The following compiler settings were used:

\begin{center}
\begin{tabular}{ll}
\toprule
Version & Flags \\
\midrule
v1 & \texttt{-O0 -g} \\
v2 & \texttt{-O3 -march=native -ffast-math} \\
v3--v5 & \texttt{-O3 -march=native -ffast-math -fopenmp} \\
\bottomrule
\end{tabular}
\end{center}

All versions were compiled with \texttt{gcc} (module \texttt{gcc/11}).

\paragraph{Runtime settings.}
OpenMP threads were pinned to cores using
\texttt{OMP\_PROC\_BIND=close} and \texttt{OMP\_PLACES=cores}.
The number of threads $T$ was varied through
\texttt{OMP\_NUM\_THREADS}.

\paragraph{Methodology.}
Matrix sizes $n \in \{2000, 4000, 6000, 8000\}$ were used to cover a range
of problem sizes.
Smaller matrices are more likely to benefit from cache reuse, while larger
matrices place more pressure on the memory system.

Thread counts $T \in \{1,2,4,8,16,32,48,64,76\}$ were used to study
strong scaling from a single core to the full node.

Each configuration was run at least three times, and the mean runtime is
reported with standard deviation error bars.
In a small number of cases, variability is higher, but this does not affect
the relative performance of the implementations.

Performance is reported in GFLOP/s using the leading-order Cholesky cost,
\[
\mathrm{GFLOP/s} = \frac{n^3}{3t \times 10^9},
\]
where $t$ is the measured runtime.
All timings include only the factorisation routine; setup and validation
are excluded.

The benchmark program records the actual OpenMP thread count at runtime,
so reported values reflect the effective number of threads used.

All figures in the report are generated from CSV data in \texttt{results/}
using the provided plotting scripts.

\section{Discussion}

\paragraph{Scaling behaviour.}
The limited scaling of v3 is consistent with two factors:
a small amount of serial work in each iteration, and frequent
synchronisation.

In the flat algorithm, each column requires a barrier.
As the number of threads increases, the work per thread per step
decreases, while the cost of synchronisation remains roughly constant.
This makes the overhead more visible at high thread counts.

The blocked algorithm (v4) reduces the number of synchronisation points,
which is consistent with the improved scaling observed in the results.

\paragraph{Super-linear speedup.}
At small thread counts, v3 shows slightly super-linear speedup.

This behaviour is often associated with improved cache usage:
as more threads are used, each thread works on a smaller portion of the
data, which may fit better in cache.

The effect is relatively small and not consistent across all cases,
so it should be interpreted cautiously.

\paragraph{Efficiency comparison (v4 vs v5).}
v5 shows lower parallel efficiency than v4, but achieves higher
absolute performance.

This is mainly because efficiency is measured relative to the
single-thread baseline. Since v5 has higher single-thread performance,
its efficiency appears lower even though it is faster at all thread
counts.

For this reason, absolute runtime or GFLOP/s provides a clearer
comparison between versions.

\paragraph{Effect of problem size.}
Performance decreases as the matrix size increases.

For smaller matrices, the data fits more easily in cache, and
throughput is higher. For larger matrices, more data must be moved
through memory, which is consistent with the lower performance observed.

This suggests that the implementation becomes more limited by memory
bandwidth at larger problem sizes.

\section{Conclusion}

Starting from a simple baseline implementation, a series of
optimisations were applied to improve both single-thread and
multi-thread performance.

Basic compiler optimisations and loop restructuring provided a large
improvement on a single core.
Introducing OpenMP parallelism gave further gains, although scaling
was limited in the flat version due to frequent synchronisation.

Rewriting the algorithm in a blocked form improved scalability by
reducing the number of synchronisation points.
Additional changes aimed at improving data locality and loop structure
further increased performance.

The final implementation reaches 182.7~GFLOP/s at 76 threads for
$n=8000$ on a CSD3 icelake node.

For larger matrices, performance appears to be limited more by memory
access than by computation.
